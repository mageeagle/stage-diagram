# Plan: Implement Group Selection Functionality (Broken Down)

This plan outlines the steps to implement group selection, copy, delete, and property updates for multiple selected nodes, while maintaining differentiated UI behavior for single vs. multiple selection. The plan is broken into smaller, manageable phases to prevent complex state issues.

## Phase 1: Core State & Actions (`useStore.ts`)

### 1.1 Update Selection State
- Replace `selectedNodeId: string | null` with `selectedNodeIds: string[]`.
- Implement `setSelectedNodeIds: (nodeIds: string[]) => void`.

### 1.2 Implement Multi-Node Deletion
- Update `deleteNode` to `deleteNodes(nodeIds: string[])`.
- Ensure it calls `recordHistory()`.
- Filter out all nodes in `nodeIds` from `nodes`.
- Filter out all edges connected to any node in `nodeIds` from `edges`.
- Clear `selectedNodeIds`.

### 1.3 Implement Multi-Node Copying
- Update `copyNode` to `copyNodes(nodeIds: string[])`.
- Ensure it calls `recordHistory()`.
- For each `nodeId` in `nodeIds`, create a new node with a new `nanoid()` and an offset position.
- Add all new nodes to the `nodes` array.

### 1.4 Implement Multi-Node Property Updates
- Refactor `updateNodeType`, `updateNodeLocation`, and `updateNodePower` to accept `nodeIds: string[]`.
- Ensure they call `recordHistory()`.
- For each `nodeId` in `nodeIds`, update the corresponding property in its `data`.

## Phase 2: Interaction & Canvas (`DiagramCanvas.tsx`)

### 2.1 Synchronize Selection
- Add `onSelectionChange` to the `ReactFlow` component to update `selectedNodeIds` in the store.

### 2.2 Multi-Node Keyboard Shortcuts
- Update the `handleKeyDown` event listener.
- Use `selectedNodeIds` instead of `selectedNodeId` for `Delete`/`Backspace` and `C` (copy) keys.

## Phase 3: Property Inspector - Single Node Mode

### 3.1 Single Node Compatibility
- Refactor `PropertyInspector.tsx` to use `selectedNodeIds`.
- Ensure that when `selectedNodeIds.length === 1`, the component still displays all properties (Label, Power, Type, Location, Inputs, Outputs) for that single selected node, maintaining existing behavior.

## Phase 4: Property Inspector - Multi-Node Mode

### 4.1 Conditional UI Implementation
- When `selectedNodeIds.length > 1`:
  - Display ONLY Type, Location, and Power.
  - The displayed values for these properties will be taken from the first node in `selectedNodeIds`.

### 4.2 Multi-Node Property Controls
- Connect the Type, Location, and Power controls in the `PropertyInspector` to the updated multi-node functions in `useStore`.

### 4.3 Group Action Buttons
- Add "Copy" and "Delete" buttons at the bottom of the `PropertyInspector` when multiple nodes are selected.

## Phase 5: Verification & Undo/Redo

### 5.1 Undo/Redo Verification
- Verify that multi-node actions (Delete, Copy, Property Updates) are correctly recorded in the history and can be undone/redone.

### 5.2 Comprehensive Testing
- Verify single-node selection works as before.
- Verify multi-node selection (via drag selection or shift+click if applicable) works correctly.
- Verify that multi-node property updates affect all selected nodes.
- Verify that Label, Inputs, and Outputs are hidden when multiple nodes are selected.
