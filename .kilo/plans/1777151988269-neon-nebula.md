**Additional Adjustment Required:**
For groups with children that have rendered content affecting their dimensions, we need to wait for the initial render to get the measured values. This requires a two-pass approach:
1. First, calculate group bounds using default sizes (current behavior)
2. On first render, let children render and populate `measured` properties
3. On subsequent renders (triggered by `groupNodesTick`), recalculate using actual `measured` values

## Trade-offs

1. **Initial sizing vs. Accuracy**: On first render, groups will use zero width/height for children without measured values. This allows them to render. On subsequent calculations, they expand to proper size.

2. **No Fallback to Hardcoded Defaults**: The current code falls back to 200x100 hardcoded sizes. The proposed fix uses 0 as the fallback, which could result in initially too-small groups. However, as nodes re-render with measured values, groups expand automatically.

3. **Two-Phase Sizing**: The group will start small and then grow. This may cause a visual "jitter" or "pop" effect. If this is a concern, we could maintain the current fallback sizes (`200` and `100`) and then expand on subsequent calculations.

## Recommendation

Use the fallback sizes in the formula:
- Replace `(n.measured?.width || 0)` with `(n.measured?.width || n.width || 200)`
- Replace `(n.measured?.height || 0)` with `(n.measured?.height || n.height || 100)`

This ensures:
1. If `measured` is available, use it (accurate sizing)
2. If `width`/`height` is available, use it (standard case)
3. Fall back to hardcoded defaults only if neither is available (degenerate case)
